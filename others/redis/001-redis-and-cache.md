# Redis与缓存

Redis一个很常见的用法就是分布式的缓存，在**旅游商城**项目中，我们用Redis来缓存已登录用户的部分信息。

## 基本流程

1. 业务线程从Redis取缓存
2. 缓存命中，直接返回
3. 缓存未命中，查数据库
4. 更新缓存
5. 返回数据

## 缓存穿透

业务现成反复请求不存在的值时，缓存总是miss，因此总是访问数据库，导致频繁查询数据库，数据库压力过大。

常见解决方式包括：

### 空值缓存

往Redis中给不存在的值保存一个空值`null`。

但缺点为：

1. 每个请求都会走到缓存，缓存压力过大
2. 存在一致性问题

### 布隆过滤器

把一般流程的第一步该为

1. 业务线程用请求的key向布隆过滤器查询
2. 数据存在，则布隆过滤器返回true，到缓存读取数据，即一般流程的1
3. 数据不存在，则布隆过滤器返回false，进入到一般流程的3
4. 在返回数据前，更新布隆过滤器的值

布隆过滤器内保存了缓存数据是否存在的信息，可以使用bitmap存储：如果数据存在，则`store[hash(key)] = 1`，否则为0。

在生产中，布隆过滤器能根据预期存储元素数`n`和可接受的误判率`r`计算需要多少个哈希函数，从而降低哈希冲突的概率，一种常见的计算方式为：

$$
k = \frac{m}{n}ln2
$$

其中`m`为需要的长整数数量：

$$
m = - \frac{nlnr}{(ln2)^2}
$$

布隆过滤器只能添加，不能删除（同一位可能对应多个数据）。

#### Redis Stack(RedisBloom)

Redis以模块的形式提供了一个内置的布隆过滤器，这里不深入。

## 缓存击穿

在某一key过期的瞬间，同一产生大量该key的请求，导致高频查询数据库。

### 互斥锁

给`查数据库 - 更新缓存`的过程加上互斥锁，只有一个人可以获取互斥锁，其他人必须等到。

一个假想的场景为：

1. 线程A查询缓存，未命中，获得锁
2. 线程B查询缓存，未命中，等待
3. 线程A再次查缓存，未命中，查数据库 - 更新缓存，释放锁
4. 线程B获得锁，查缓存命中，释放锁

这里获得锁的一方还需要再查一次缓存，从而避免所有线程都要查询一次数据库的情况。

这一方案牺牲了可用性，保证了一致性。

### 逻辑过期时间

不再给key设置过期时间，而是在值中保存一个逻辑过期时间。

一个假想的场景为：

1. 线程A查询缓存，发现已经逻辑过期，锁上互斥锁，发起新线程B，返回已过期的旧数据
2. 线程C查询缓存，发现已经逻辑过期，互斥锁已锁，直接返回已过期的旧数据
3. 线程B更新数据，释放互斥锁
4. 线程D查询缓存，发现没有逻辑过期，返回新数据

这一方案牺牲了一致性，保证了可用性。

### singleflight

这是go语言提供的独特解决方案：`golang.org/x/sync/singleflight`。

`singleflight`类似于`monitor goroutine`，即只允许一个goroutine访问缓存，但存在不同。

`monitor goroutine`是通过channel实现的，只能串行处理任务。而`singleflight`实现了请求合并，它内部维护一个map，保存了key和请求该key的goroutine的信息，在第一次接收到一个key的请求时去执行，后续所有对同一key的请求全部阻塞，当该key处理完成，则返回给所有请求该key的goroutine，可以说是“并行”的。
