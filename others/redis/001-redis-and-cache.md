# Redis与缓存

Redis一个很常见的用法就是分布式的缓存，在**旅游商城**项目中，我们用Redis来缓存已登录用户的部分信息。

## 基本流程

1. 业务线程从Redis取缓存
2. 缓存命中，直接返回
3. 缓存未命中，查数据库
4. 更新缓存
5. 返回数据

## 缓存穿透

业务现成反复请求不存在的值时，缓存总是miss，因此总是访问数据库，导致频繁查询数据库，数据库压力过大。

常见解决方式包括：

### 空值缓存

往Redis中给不存在的值保存一个空值`null`。

但缺点为：

1. 每个请求都会走到缓存，缓存压力过大
2. 存在一致性问题

### 布隆过滤器

把一般流程的第一步该为

1. 业务线程用请求的key向布隆过滤器查询
2. 数据存在，则布隆过滤器返回true，到缓存读取数据，即一般流程的1
3. 数据不存在，则布隆过滤器返回false，进入到一般流程的3
4. 在返回数据前，更新布隆过滤器的值

布隆过滤器内保存了缓存数据是否存在的信息，可以使用bitmap存储：如果数据存在，则`store[hash(key)] = 1`，否则为0。

在生产中，布隆过滤器能根据预期存储元素数`n`和可接受的误判率`r`计算需要多少个哈希函数，从而降低哈希冲突的概率，一种常见的计算方式为：

$$
k = \frac{m}{n}ln2
$$

其中`m`为需要的长整数数量：

$$
m = - \frac{nlnr}{(ln2)^2}
$$

布隆过滤器只能添加，不能删除（同一位可能对应多个数据）。

#### Redis Stack(RedisBloom)

Redis以模块的形式提供了一个内置的布隆过滤器，这里不深入。

## 缓存击穿

在某一key过期的瞬间，同一产生大量该key的请求，导致高频查询数据库。

### 互斥锁

给`查数据库 - 更新缓存`的过程加上互斥锁，只有一个人可以获取互斥锁，其他人必须等到。

一个假想的场景为：

1. 线程A查询缓存，未命中，获得锁
2. 线程B查询缓存，未命中，等待
3. 线程A再次查缓存，未命中，查数据库 - 更新缓存，释放锁
4. 线程B获得锁，查缓存命中，释放锁

这里获得锁的一方还需要再查一次缓存，从而避免所有线程都要查询一次数据库的情况。

这一方案牺牲了可用性，保证了一致性。

### 逻辑过期时间

不再给key设置过期时间，而是在值中保存一个逻辑过期时间。

一个假想的场景为：

1. 线程A查询缓存，发现已经逻辑过期，锁上互斥锁，发起新线程B，返回已过期的旧数据
2. 线程C查询缓存，发现已经逻辑过期，互斥锁已锁，直接返回已过期的旧数据
3. 线程B更新数据，释放互斥锁
4. 线程D查询缓存，发现没有逻辑过期，返回新数据

这一方案牺牲了一致性，保证了可用性。

### singleflight

这是go语言提供的独特解决方案：`golang.org/x/sync/singleflight`。

`singleflight`类似于`monitor goroutine`，即只允许一个goroutine访问缓存，但存在不同。

`monitor goroutine`是通过channel实现的，只能串行处理任务。而`singleflight`实现了请求合并，它内部维护一个map，保存了key和请求该key的goroutine的信息，在第一次接收到一个key的请求时去执行，后续所有对同一key的请求全部阻塞，当该key处理完成，则返回给所有请求该key的goroutine，可以说是“并行”的。

## 缓存雪崩

大量key同时过期或Redis崩溃导致大量访问数据库。

1. 随机TTL：可以给TTL增加随机的偏移量，避免有多个key共享相同的有效期
2. Redis集群：部署多个Redis，避免因为Redis崩溃引起的问题
3. 降级限流
4. 多级缓存

## 缓存的双写一致

1. 读：缓存命中，直接返回；缓存未命中，查数据库，写入缓存，设定超时时间；
2. 写：
  - 【延迟双删】先删缓存，修改数据库，再删延时（DB可能需要主从复制）缓存（避免前面两个操作之间，有另一个线程往缓存写旧数据）
  - 【读写锁】加锁可以保证绝对的一致性，但是性能太差
  - 【异步通知】通过MQ发布更新消息，保证只有一个现成在写线程
  - 【canal】阿里的中间件，直接监听MySQL的binlog，直接写缓存（不涉及业务代码）

## 缓存的持久化

为了避免宕机导致数据丢失，Redis内部有两种数据持久化的方式：RDB(Redis DataBase)和AOF(Append Only File)。

### RDB (Redis Database)

RDB是Redis 默认的持久化方案。它在指定的时间间隔内将内存中的全量数据写入磁盘。

- 优点：生成的是紧凑的二进制文件，其速度远快于 AOF
- 缺点：
  1. 由于是间隔触发，如果 Redis 意外宕机，最后一次快照之后的数据将会丢失
  2. 在大数据量下，fork 子进程进行快照操作可能会导致服务器瞬间的性能抖动

### AOF (Append Only File)

AOF通过记录Redis执行的每一条写指令来实现，他们被Append到一个文件中。Redis启动时会读取该文件并重新执行这些命令来重建数据。

- 优点：
  1. 更好的安全性
  2. 可以通过配置把数据持久化设置到秒级
- 缺点：随着运行时间增长，AOF文件会变得十分巨大，导致恢复速度远慢于RDB

### RDB 与 AOF 对比

| 维度 | RDB | AOF |
| --- | --- | --- |
| **持久化方式** | 定时对整个内存做快照 | 记录每一次执行的命令 |
| **数据完整性** | 不完整，两次备份之间会丢失 | 相对完整，取决于刷盘策略 |
| **文件大小** | 会有压缩，文件体积小 | 记录命令，文件体积很大 |
| **宕机恢复速度** | 很快 | 慢 |
| **数据恢复优先级** | 低，因为数据完整性不如 AOF | 高，因为数据完整性更高 |
| **系统资源占用** | 高，大量 CPU 和内存消耗 | 低，主要是磁盘 IO 资源，但 AOF 重写时会占用大量 CPU 和内存资源 |
| **使用场景** | 可以容忍数分钟的数据丢失，追求更快的启动速度 | 对数据安全性要求较高常见 |

---

### 使用场景

现在最主流的方案是 **混合持久化 (RDB + AOF)**。

- **原理**：在 AOF 重写时，将当前内存数据以 RDB 形式写入 AOF 文件头部，后续增量命令再以 AOF 格式追加。
- **优点**：结合了 RDB 的**加载快**和 AOF 的**丢失少**。

## 数据过期策略

给Key设置TTL后，这个Key就会在某个时刻过期，Redis会删除这些Key和对应的数据，包括两种删除策略：惰性删除和定期删除。

### 惰性删除

只在访问Key时检查是否过期并删除。

优点：对CPU友好
缺点：对内存不友好

### 定期删除

按照一定的频率扫描所有的Key检查是否过期并删除。

#### Slow模式

默认按照10hz的频率扫描，但每次扫描不超过25ms。

#### Fast模式

在事件循环中总是尝试扫描，但是两次扫描间隔不低于2ms，单次扫描事件不超过1ms。

优点：可配置的扫描频率和时间
缺点：对CPU不友好

## 数据淘汰策略 maxmemory-policy

- noeivtion
- volatile-ttl：淘汰马上过期的
- allkeys-random
- volatile-random
- allkeys-lru
- volatile-lru
- allkeys-lfu
- volatile-lfu

allkeys对应所有key，volatile对应有ttl的key。random表示随机淘汰，LRU表示最近最少访问，LFU表示最近访问频率最低。

如何选择，还是看业务逻辑。在`旅游商城`中，Redis被用来保存用户信息，在每次通过gateway做权限校验时都需要查询一遍，因此我们采用LRU策略，把最近最长访问的数据保存在缓存里。
