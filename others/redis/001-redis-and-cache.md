# Redis与缓存

Redis一个很常见的用法就是分布式的缓存，在**旅游商城**项目中，我们用Redis来缓存已登录用户的部分信息。

## 基本流程

1. 业务线程从Redis取缓存
2. 缓存命中，直接返回
3. 缓存未命中，查数据库
4. 更新缓存
5. 返回数据

## 缓存穿透

业务现成反复请求不存在的值时，缓存总是miss，因此总是访问数据库，导致频繁查询数据库，数据库压力过大。

常见解决方式包括：

### 空值缓存

往Redis中给不存在的值保存一个空值`null`。

但缺点为：

1. 每个请求都会走到缓存，缓存压力过大
2. 存在一致性问题

### 布隆过滤器

把一般流程的第一步该为

1. 业务线程用请求的key向布隆过滤器查询
2. 数据存在，则布隆过滤器返回true，到缓存读取数据，即一般流程的1
3. 数据不存在，则布隆过滤器返回false，进入到一般流程的3
4. 在返回数据前，更新布隆过滤器的值

布隆过滤器内保存了缓存数据是否存在的信息，可以使用bitmap存储：如果数据存在，则`store[hash(key)] = 1`，否则为0。

在生产中，布隆过滤器能根据预期存储元素数`n`和可接受的误判率`r`计算需要多少个哈希函数，从而降低哈希冲突的概率，一种常见的计算方式为：

$$
k = \frac{m}{n}ln2
$$

其中`m`为需要的长整数数量：

$$
m = - \frac{nlnr}{(ln2)^2}
$$

布隆过滤器只能添加，不能删除（同一位可能对应多个数据）。

#### Redis Stack(RedisBloom)

Redis以模块的形式提供了一个内置的布隆过滤器，这里不深入。
